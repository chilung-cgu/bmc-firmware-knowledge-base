# 為什麼需要 MCTP？TCP/IP 不夠嗎？

本文詳細解釋 MCTP（Management Component Transport Protocol）存在的理由、它與 TCP/IP 的定位差異，以及硬體設計如何決定底層實體匯流排（I2C, PCIe, USB）的選擇。

---

## 1. 為什麼不能直接用 TCP/IP？

這是一個核心問題。簡短的答案是：**TCP/IP 是為「網路世界」設計的，而 MCTP 是為「平台內部管理世界」設計的，兩者解決的問題完全不同。**

### 核心差異：目標場景不同

| 面向 | TCP/IP | MCTP |
|------|--------|------|
| **設計目標** | 跨網路（LAN/WAN）的通用資料通訊 | 伺服器/平台**內部**元件之間的**管理通訊** |
| **通訊對象** | 電腦 ↔ 電腦 | BMC ↔ CPU、GPU、NVMe SSD、PSU 等 |
| **網路規模** | 數十億節點的全球網路 | 同一台機器內的幾十個元件 |
| **端點能力** | 有完整 OS 和充足 RAM/CPU | 微控制器、韌體、幾 KB RAM |

### 為什麼嵌入式環境不適合 TCP/IP？

#### 1. 嵌入式元件跑不動 TCP/IP

一顆 PSU（電源供應器）的微控制器可能只有 **幾 KB 的 RAM**。TCP/IP 完整的協議堆疊（ARP、IP、TCP、socket buffer management、擁塞控制等）對它來說太重了。

MCTP 的設計非常精簡：
- Header 只有 **4 bytes**（Version + Dest EID + Src EID + Tag）
- EID 只有 **8-bit**（最多 255 個端點，對平台內管理足夠）
- **無連接**、**無擁塞控制**——因為在同一台機器內部不需要

#### 2. TCP/IP 無法直接跑在 I2C、PCIe VDM 這些匯流排上

平台內部元件之間的連接方式是：

- **BMC ↔ SSD**：走 **I2C/SMBus**
- **BMC ↔ GPU/加速器**：走 **PCIe VDM** (Vendor Defined Messages)
- **BMC ↔ 其他管理控制器**：走 **USB** 或 **Serial**

TCP/IP 需要底層有 Ethernet 或類似的 packet-based 網路介面。你不能把 TCP/IP 直接塞進 I2C（一個低速的序列匯流排）或 PCIe VDM（一種 TLP 類型）中——技術上雖非完全不可能，但**極不實際且完全沒必要**。

MCTP 的關鍵設計就是 **Transport Binding**（傳輸綁定）：

```
上層協議（PLDM, SPDM, NVMe-MI）
         │
       MCTP（統一傳輸層）
         │
    ┌────┼────┐
    I2C  PCIe  USB    ← 不同的 Physical Binding
```

#### 3. TCP/IP 的定址模型不適合

- TCP/IP 用 **32/128-bit IP + 16-bit Port** 定址：對一個只有數個端點的平台內管理網路來說太浪費
- MCTP 用 **8-bit EID**：簡單、輕量、由 Bus Owner（通常是 BMC）集中分配
- 且 EID 與物理地址（I2C slave address、PCIe BDF）**解耦**，讓上層協議不需關心底層用什麼匯流排

#### 4. 歷史背景：取代 IPMI 的碎片化

在 MCTP 之前，平台管理通訊是 **碎片化** 的：
- 和 PSU 通訊用 PMBus（over I2C）
- 和 CPU 通訊用 IPMI/KCS
- 和 NVMe SSD 通訊用各家自訂協議
- 和 GPU 通訊也各家不同

每種設備有自己的協議，上層管理軟體要為每種都寫不同的驅動。**MCTP + PLDM 的組合就是要統一這一切：**

> **一個標準的傳輸層（MCTP）+ 一個標準的管理語義層（PLDM）= 不管底層匯流排是什麼、不管對面是什麼設備，都用同一套協議溝通。**

### 類比：網際網路 vs 平台管理

| 層次 | 網際網路世界 | 平台管理世界 |
|------|------------|-------------|
| 應用層 | HTTP, DNS, SMTP | **PLDM**, SPDM, NVMe-MI |
| 傳輸層 | TCP, UDP | **MCTP** |
| 實體/鏈路層 | Ethernet, Wi-Fi | **I2C, PCIe VDM, USB, Serial** |

MCTP 就是平台管理世界的「TCP/UDP」——但為了嵌入式環境做了大幅精簡。

## 2. 實體/鏈路層的選擇：I2C, PCIe VDM, USB

為什麼硬體溝通會選擇特定的匯流排？重點是什麼？速度嗎？

### 核心觀念：不是「選擇」，而是「已經存在」

這些物理層 **不是 MCTP 發明的**，它們是硬體設計時 **早就焊在主機板上的匯流排**。MCTP 只是「借用」這些既有的線路來傳管理訊息。

> **重點：不是「我要選哪條匯流排」，而是「這個設備本來就接在哪條匯流排上」。**
> **選擇哪種 physical binding 不是由軟體決定，而是由硬體設計（那個設備接在哪條匯流排上）決定。MCTP 的價值就是讓上層軟體不需要關心這個差異。**

### 硬體工程師的選擇邏輯

硬體工程師在設計電路板時，為什麼會選 I2C 而不是 PCIe？選擇的考量遠不只速度。

> **硬體工程師的選擇邏輯：用最便宜、剛好夠用的匯流排。不會為了讀溫度去接 PCIe，也不會要 GPU 走 I2C。**

### 匯流排選擇的關鍵考量因素 (6 個面向)

#### 1️⃣ 成本（最重要的因素之一）

| 匯流排 | 線路數 | 接腳/連接器成本 | 說明 |
|--------|--------|----------------|------|
| **I2C** | **2 條**（SDA + SCL） | 極低 | 便宜到幾乎免費，所以低價元件都用它 |
| **SPI** | 4+ 條 | 低 | 比 I2C 快但多幾條線 |
| **USB** | 4 條（2 data + 2 power） | 中 | 需要專用連接器 |
| **PCIe** | 很多（x1=4, x16=64 差分對） | 高 | 需要高品質 PCB 走線、專用插槽 |
| **Ethernet** | 4-8 條 | 中-高 | 需要 PHY 晶片、RJ45 連接器、變壓器 |

> **一顆 ¥5 的溫度感測器不會配一個 ¥50 的 PCIe 介面。**
> 所以低價感測器、PSU、EEPROM 統統用 I2C——因為只要 **2 條線**。

#### 2️⃣ 速度 vs 需求匹配 (Speed vs. Demand)

| 匯流排 | 典型速度 | 適合什麼 |
|--------|---------|---------|
| **I2C** | 100 KHz ~ 1 MHz | 讀溫度、電壓、設定暫存器（幾 bytes） |
| **SPI** | 1 ~ 100 MHz | Flash 晶片讀寫、小型顯示器 |
| **USB 2.0** | 480 Mbps | 鍵盤滑鼠、中速資料傳輸 |
| **PCIe 4.0 x16** | ~32 GB/s | GPU、NVMe SSD（大量資料搬運） |
| **Ethernet 1G** | 1 Gbps | 網路通訊 |

> **關鍵：不是越快越好，而是夠用就好。** 讀一個溫度值只需要 2 bytes，用 PCIe 的 32 GB/s 去讀完全是殺雞用牛刀。

#### 3️⃣ 拓撲架構（幾個設備？怎麼接？）

| 匯流排 | 拓撲 | 一條 bus 上多少設備 |
|--------|------|------------------|
| **I2C** | **多主多從共享 bus** | 一條 bus 可掛 ~10-100 個設備 |
| **SPI** | 一主多從（每個從需要一條 CS 線） | 受限於 CS 線數量 |
| **USB** | 星狀（Hub 擴展） | 最多 127 個 |
| **PCIe** | 點對點 + Switch 擴展 | 每個 slot 一個設備 |

> **I2C 一條 bus 可以掛很多低速設備**，非常適合主機板上散佈各處的感測器。
> **PCIe 是點對點的**，每個設備獨享頻寬，適合高速設備。

#### 4️⃣ 電氣複雜度與 PCB 設計

| 匯流排 | PCB 要求 | 說明 |
|--------|---------|------|
| **I2C** | 極簡單 | 兩條線拉一拉就好，不需要特殊阻抗控制 |
| **SPI** | 簡單 | 幾條線，短距離即可 |
| **USB** | 中等 | 差分對需要 90Ω 阻抗匹配 |
| **PCIe** | 複雜 | 高速差分對、嚴格等長走線、多層 PCB |

> **PCIe 走線在 PCB 上非常「嬌貴」**——要精確的阻抗控制、等長匹配、避免噪訊干擾。硬體成本和設計難度都高。

#### 5️⃣ 距離

| 匯流排 | 典型距離 | 說明 |
|--------|---------|------|
| **I2C** | < 1 公尺 | 同一塊板子上 |
| **USB** | < 5 公尺 | 可接線到外部 |
| **PCIe** | < 30 公分（板上）| 主要在主機板內部或透過 riser card |
| **Ethernet** | < 100 公尺（銅纜） | 跨機器、跨機櫃 |

> **Ethernet 可以跨樓層通訊，I2C 連跨板子都勉強。** 不同場景需要不同距離。

#### 6️⃣ CPU/MCU 資源消耗

| 匯流排 | 端點硬體需求 | 說明 |
|--------|-------------|------|
| **I2C** | 極少（幾百 gate 的邏輯） | 最便宜的 MCU 都有 |
| **PCIe** | 高（需要 PCIe PHY + 控制器） | 通常是高端 IC 才有 |
| **Ethernet** | 高（需要 MAC + PHY） | 需要較多邏輯資源 |

## 3. 圖解：為什麼不同設備用不同匯流排

```
                         速度需求
                           ↑
                           │
              PCIe ════════╪══════ 高速 + 高成本
              (GPU, SSD)   │       (本來就插在 PCIe 槽)
                           │
              USB ─────────│───── 中速 + 通用
              (多節點)      │       (BMC 間通訊)
                           │
              SPI ─────────│───── 中低速 + 簡單
              (Flash)      │
                           │
              I2C ═════════╪══════ 低速 + 極低成本
              (感測器,PSU)  │       (最普遍的選擇)
                           └──────────────────→ 成本/複雜度
                          低                      高
```

### 實際例子：一台 AI 伺服器

```
┌──────────────────────────────────────────────────────┐
│                    AI 伺服器                          │
│                                                      │
│  BMC (AST2600)                                       │
│   │                                                  │
│   ├── I2C bus 0 ── PSU #1         (MCTP over I2C)    │
│   ├── I2C bus 0 ── PSU #2         (MCTP over I2C)    │
│   ├── I2C bus 2 ── Temp Sensors   (MCTP over I2C)    │
│   ├── I2C bus 5 ── BIC (Bridge)   (MCTP over I2C)    │
│   │                  │                               │
│   │                  ├── CPU                         │
│   │                  └── DIMM Sensors                │
│   │                                                  │
│   └── PCIe ──── GPU #1 (NVIDIA)   (MCTP over PCIe)   │
│          ├──── GPU #2 (NVIDIA)     (MCTP over PCIe)   │
│          ├──── NVMe SSD #1         (MCTP over PCIe)   │
│          └──── NVMe SSD #2         (MCTP over PCIe)   │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**結論**：
MCTP 的高明之處在於，它不強迫硬體改變（PSU 繼續用便宜的 I2C，GPU 繼續用高速的 PCIe），而是提供一個**軟體抽象層**。上層的 PLDM 程式碼寫一次，就可以同時管理 I2C 上的電源和 PCIe 上的 GPU。
